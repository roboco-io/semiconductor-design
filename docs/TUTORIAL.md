# 튜토리얼 — AI 에이전트가 "반도체 타이밍 예측 모델"을 스스로 연구하게 만들기

> 이 문서는 **반도체(EDA) 배경 지식이 전혀 없는 사람**도 이 프로젝트가 무엇을, 왜, 어떻게
> 하는지 이해할 수 있도록 용어를 하나하나 풀어 쓴 튜토리얼입니다. 끝에 [용어 사전](#용어-사전)이 있습니다.

---

## 0. 한 문단 요약

칩(반도체)을 설계할 때는 "이 회로가 충분히 빠른가?"를 확인하려고 **무겁고 느린 시뮬레이션**을
돌립니다. 이걸 매번 돌리는 대신, 빠르게 결과를 *예측*하는 **AI 모델(surrogate, 대리 모델)** 을
만들 수 있습니다. 문제는 그 AI 모델을 잘 만드는 일 자체가 사람이 손으로 이것저것 시도하는
**지루한 반복 노동**이라는 점입니다. 이 프로젝트는 그 반복을 **AI 에이전트에게 맡겨**
자동으로 더 나은 예측 모델을 찾게 합니다. 단, **최종 채택은 항상 사람이 승인**합니다.

---

## 1. 먼저, 칩은 어떻게 만들어지나? (5분 속성 과외)

칩(예: CPU)은 수억 개의 **트랜지스터**(아주 작은 전자 스위치)로 된 회로입니다. 사람이 그걸
하나하나 그리는 건 불가능하므로, **여러 자동화 단계**를 거쳐 만듭니다. 건물 짓기에 비유하면:

| 단계 | 하는 일 | 건물 비유 | 도구(오픈소스) |
|---|---|---|---|
| **RTL 작성** | 회로의 동작을 코드로 기술 (Verilog 언어) | 건축 설계도(개념) | 사람이 작성 |
| **합성(Synthesis)** | 코드 → 실제 논리 게이트(AND/OR/플립플롭) 목록으로 변환 | 설계도 → 자재 명세서 | **Yosys** |
| **배치&배선(Place & Route)** | 게이트를 칩 평면에 배치하고 전선으로 연결 | 자재를 실제로 배치·배관 | **OpenROAD** |
| **사인오프(Sign-off)** | "진짜 동작하나?" 최종 검증 | 준공 검사 | OpenSTA 등 |
| **GDSII** | 공장에 보낼 최종 도면 | 시공용 최종 도면 | — |

이 전체 자동화 흐름을 **EDA**(Electronic Design Automation, 전자 설계 자동화)라고 부릅니다.
이 프로젝트는 **OpenROAD-flow-scripts(ORFS)** 라는 오픈소스 EDA 흐름을 사용합니다.

### 우리가 관심 있는 한 가지 숫자: "타이밍 슬랙(timing slack)"

칩 안에서 신호는 **클럭(clock)** 박자에 맞춰 한 칸씩 이동합니다. 한 박자(예: 1.1 나노초) 안에
신호가 출발점에서 도착점까지 못 가면 칩이 **오작동**합니다.

- **타이밍 슬랙** = "마감 시간(한 박자)까지 남은 여유 시간".
  - 슬랙이 **+면 여유 있음**(좋음), **−면 마감 초과**(나쁨 = 회로가 너무 느림).
- 신호가 지나는 길 하나를 **path(경로)**, 그 길의 도착점(플립플롭의 입력 등)을
  **endpoint(종점)** 라고 합니다.

> 비유: 택배가 마감 시각까지 배송돼야 하는데, "마감까지 남은 시간"이 슬랙입니다.
> 음수면 이미 늦은 것.

이 슬랙을 알려면 **STA**(Static Timing Analysis, 정적 타이밍 분석)라는 계산을 돌리고,
그 결과를 `report_checks`라는 **텍스트 리포트**로 뽑습니다. 우리 시스템이 읽는 게 바로 이 리포트입니다.

## 2. "대리 모델(surrogate)"이란?

진짜 슬랙을 알려면 배치&배선까지 **끝까지 돌려야** 하는데, 이건 **느립니다**(작은 칩도 수 분,
큰 칩은 몇 시간~며칠). 그래서 연구자들은 이런 아이디어를 씁니다:

> **합성 직후의 빠른 정보(feature)** 만 보고 **최종 슬랙(label)** 을 *예측*하는 **AI 모델**을
> 만들자. 그러면 끝까지 안 돌려도 결과를 미리 가늠할 수 있다.

이 "느린 진짜 계산을 흉내 내는 빠른 AI 모델"을 **surrogate model(대리 모델)** 이라고 합니다.

- **feature(입력 특징)**: 합성 직후 각 경로의 정보 — 게이트 단 수, 합성 단계 슬랙 등 (총 8개).
- **label(정답)**: 그 경로의 *최종*(배선 후) 슬랙 — 우리가 맞히려는 값.
- **모델**: feature → label을 예측하는 함수. 우리는 **tabular(표 형태 데이터) 머신러닝**,
  구체적으로 **sklearn의 Gradient Boosting**(결정 트리 여러 개를 합치는 방법)을 씁니다.
  (이미지/그래프 신경망이 아니라 *표 데이터에 강한 트리 모델*을 고른 이유는 [용어 사전](#왜-gnn이-아니라-tabular-트리-모델인가) 참고.)
- **잘함의 기준**: **MAE**(Mean Absolute Error, 평균 절대 오차) — 예측이 정답에서 평균
  몇 나노초 빗나갔나. **낮을수록 좋음.**

## 3. "AutoResearch"란? — 연구를 검색으로 바꾸기

대리 모델을 잘 만들려면 사람이 "feature를 이렇게 바꿔볼까? 트리 깊이를 늘려볼까?"를
**수없이 반복**해야 합니다. 이건 창의적이라기보다 **기계적**입니다.

karpathy의 **[AutoResearch](https://github.com/karpathy/autoresearch)** 아이디어:

> "연구"를 "검색(search)"으로 환원하자. **학습 스크립트(`train.py`) 하나**를 두고,
> **AI 에이전트가 그 스크립트를 조금씩 변형(mutation)** 하며 더 나은 버전을 자동으로 찾게 하자.

이 프로젝트는 그걸 **EDA 대리 모델**에 적용합니다. 그리고 결정적 차별점 하나를 둡니다:

> 원조 AutoResearch는 **"사람은 자고 있어도 된다(완전 자율)"** 를 전제합니다.
> 우리는 반대로 **사람(Operator)이 매 세대 winner를 승인**합니다. 이게 우리 연구의 핵심 가설입니다.

- **가설 H-A**: 에이전트 루프가 사람 수작업보다 *더 나은* 대리 모델을 *적은 노력*으로 찾는다.
- **가설 H-B**: 사람이 승인 단계를 유지해도 성능을 잃지 않으며, 오히려 신뢰성이 높아진다.

## 4. 시스템 구성 — 4개의 조각

```
 ┌──────────────────────┐   진짜 EDA 흐름을 클라우드(AWS Fargate)에서 1회 실행
 │ 1. EDA flow (Fargate)│ ──────────────────────────────────┐
 └──────────────────────┘                                   │ STA 리포트(synth/route)
                                                             ▼
 ┌──────────────────────┐   리포트 2개를 읽어 "feature+label" 표(dataset)로 변환
 │ 2. prepare.py (고정)  │ ──────────────────────────────────┐
 └──────────────────────┘                                   │ dataset.jsonl (53행)
                                                             ▼
 ┌──────────────────────┐   표를 입력받아 대리 모델 학습, 정확도(val_mae) 출력
 │ 3. train.py (변형대상)│ ◀── 에이전트가 이 파일을 변형 ──┐  │
 └──────────────────────┘                                │  │
                                                          │  ▼
 ┌────────────────────────────────────────────────────────────────┐
 │ 4. 진화 루프 (src/pipeline)                                      │
 │  후보 N개 생성(Claude+Codex) → 각각 실행 → 최저 val_mae winner    │
 │  → ★사람(Operator)이 승인 → git tag로 baseline 승격              │
 └────────────────────────────────────────────────────────────────┘
```

### 조각 1 — EDA flow (`docker/`, `cdk/`)
진짜 칩 설계 흐름(`gcd`라는 작은 예제 칩)을 **AWS Fargate**(클라우드의 임시 컴퓨터)에서 돌립니다.
왜 클라우드? 이 흐름의 한 단계(CTS)가 맥북의 가상화 환경에서 죽었기 때문입니다(아래 6장).
결과물: 합성 직후 STA 리포트 + 배선 후 STA 리포트 → S3(클라우드 저장소)에 저장.

### 조각 2 — `prepare.py` (데이터 준비, **고정/frozen**)
두 STA 리포트를 읽어 **표(dataset)** 로 만듭니다. 한 행 = 한 endpoint, 컬럼 = 8개 feature + 정답 슬랙.
이 파일은 **에이전트가 못 건드립니다**(공정한 비교를 위해 데이터·평가 규칙을 고정).
실제로 진짜 데이터에서 **53행**을 만들었습니다.

### 조각 3 — `train.py` (학습 스크립트, **에이전트가 변형**)
표를 받아 대리 모델을 학습하고 `{"val_mae": 0.17}` 같은 점수를 출력합니다.
**이 한 파일이 에이전트의 유일한 변형 대상**입니다. (sklearn/numpy만 사용, 단일 파일 규칙.)

### 조각 4 — 진화 루프 (`src/pipeline/`)
1. **candidate_gen**: Claude와 Codex(두 AI 코딩 에이전트)에게 "train.py를 이렇게 개선해봐"라고
   시켜 변형본 N개를 만듭니다.
2. **runner**: 각 변형본을 실제로 실행해 val_mae를 잰다.
3. **selection**: 가장 낮은 val_mae를 낸 후보를 winner로 고른다.
4. **operator_gate**: winner와 그 변경 내용을 **사람에게 보여주고 승인**을 받는다.
   승인해야만 그 winner가 다음 세대의 출발점(baseline)이 되고 `gen-NNN-best` 꼬리표가 붙는다.
   **승인 전까지 baseline은 절대 안 바뀝니다**(가설 H-B를 코드 구조로 강제).

## 5. 실제로 무슨 일이 일어났나? (gen-001 결과)

진짜 데이터 53행으로 **첫 세대**를 돌렸습니다.

| 모델 | val_mae (낮을수록 좋음) | 의미 |
|---|---|---|
| naive(아무것도 안 한 기준선) | 1.41 ns | "합성 슬랙을 그냥 최종 슬랙이라 치자" |
| baseline `train.py`(사람이 만든 기본) | 0.177 ns | 기본 Gradient Boosting |
| **gen-001 winner (Codex가 변형)** | **~0.11 ns** | 도메인 feature 추가 + 모델 앙상블 |

- winner는 단순히 숫자를 튜닝한 게 아니라, **타이밍에 의미 있는 새 feature**(예: "경로 임계도",
  "단 당 슬랙")를 *발명*하고 여러 모델을 **앙상블**(투표)했습니다. → 가설 **H-A 첫 확증**.
- 시스템은 winner를 **자동 채택하지 않고 멈춰** 사람에게 승인을 물었고, 사람이 승인한 뒤에야
  승격했습니다. → 가설 **H-B 확증**.

## 6. 솔직한 교훈 — "진짜로 돌려봐야 보이는 것들"

이 프로젝트에서 가장 값진 부분은 **가짜(합성) 데이터로는 절대 안 드러나는 문제들**을
*진짜 실행*이 표면화했다는 점입니다.

1. **두 시점의 critical path가 다르다**: "합성 때 가장 급한 경로"와 "배선 후 가장 급한 경로"가
   서로 달라, 단순히 경로를 짝지으면 데이터가 0행이 됐습니다. → 짝짓는 기준을 *path*가 아니라
   *안정적인 endpoint*로 바꿔 해결.
2. **로컬에서 진짜 칩 흐름이 죽었다**: 맥북(ARM)에서 x86 EDA 도구를 에뮬레이션하니 한 단계(CTS)가
   죽었습니다. → AWS Fargate의 진짜 x86에서 돌려 해결(이게 클라우드를 쓴 진짜 이유).
3. **AI가 만든 모델이 "다른 컴퓨터에서 안 열렸다"**: 에이전트가 만든 모델을 저장했다가
   다른 프로세스에서 다시 불러올 때 깨졌습니다(파이썬 pickle의 함정). → 재채점 코드를 고쳐 해결.
4. **검증 게이트는 사람에게도 필요하다**: 개발 중 제가(Claude) 테스트 실패를 놓치고 커밋해
   잠깐 main이 깨졌습니다. → 검증을 *자동으로 강제*해야 한다는 교훈(사람도 실수한다).

이런 마찰을 *발견하고 고치는 것*이 바로 AutoResearch가 존재하는 이유입니다 —
기계적이지만 사람이 놓치기 쉬운 것을 드러냅니다.

## 용어 사전

- **EDA** (Electronic Design Automation): 칩 설계를 자동화하는 소프트웨어 도구 전체.
- **RTL** (Register-Transfer Level): 회로 동작을 코드로 기술한 것. 주로 **Verilog** 언어.
- **합성(Synthesis)**: RTL 코드를 실제 논리 게이트 목록(**netlist**)으로 변환. 도구: **Yosys**.
- **게이트/플립플롭**: 게이트는 AND/OR 같은 기본 논리 소자. 플립플롭(FF)은 한 박자 동안 값을
  기억하는 소자(신호의 출발/도착점).
- **배치&배선(Place & Route)**: 게이트를 칩 평면에 놓고(placement) 전선으로 잇는(routing) 단계.
  도구: **OpenROAD**.
- **CTS** (Clock Tree Synthesis): 클럭 신호를 칩 전체에 고르게 분배하는 배선 단계.
- **STA** (Static Timing Analysis): 모든 경로의 타이밍을 계산해 "마감을 지키는가"를 검사.
  결과를 `report_checks` 리포트로 출력.
- **path / endpoint**: path는 신호가 지나는 한 경로. endpoint는 그 도착점(주로 플립플롭 입력).
- **slack(슬랙)**: "한 박자 마감까지 남은 여유 시간". +면 OK, −면 너무 느림.
- **PDK** (Process Design Kit): 특정 공정(여기선 오픈소스 **sky130**)의 게이트 라이브러리·규칙 모음.
- **ORFS** (OpenROAD-flow-scripts): 위 단계들을 한 번에 돌려주는 오픈소스 흐름.
- **surrogate model(대리 모델)**: 느린 진짜 계산을 흉내 내는 빠른 예측 AI 모델.
- **feature / label**: feature는 입력 특징, label은 맞히려는 정답.
- **MAE**: 평균 절대 오차. 예측이 정답에서 평균 얼마나 빗나갔나(낮을수록 좋음).
- **val_mae**: 검증 데이터에서 잰 MAE. 우리 루프가 **최소화**하려는 단일 점수.
- **AutoResearch**: "연구를 검색으로" — 에이전트가 학습 스크립트를 변형하며 자동 개선.
- **Operator**: 이 시스템을 감독하는 사람(당신). winner 승인 권한을 가짐.
- **Fargate / S3 / CDK**: AWS의 임시 컴퓨터 / 클라우드 저장소 / 인프라를 코드로 정의하는 도구.

### 왜 GNN이 아니라 tabular 트리 모델인가?
회로는 그래프(노드=게이트, 엣지=전선)라 **그래프 신경망(GNN)** 이 어울려 보입니다. 하지만
(1) 우리 데이터는 "경로별 8개 숫자 feature"라 **표(tabular) 형태**이고, (2) 표 데이터에서는
**Gradient Boosting 트리(XGBoost·sklearn)** 가 신경망을 자주 이긴다는 연구가 많으며
(특히 데이터가 적고 분포가 한쪽으로 쏠릴 때), (3) GNN은 GPU·무거운 라이브러리가 필요해
"CPU·가벼운 의존성" 제약과 충돌합니다. 그래서 **sklearn 트리 모델**을 골랐고,
실제로 권위 있는 선행연구(MasterRTL)도 path-level 타이밍 예측에 트리 모델을 써서 GNN을
앞섰습니다. 근거: [`docs/knowledge-base/2026-06-06-od4-model-class-research.md`](knowledge-base/2026-06-06-od4-model-class-research.md).
