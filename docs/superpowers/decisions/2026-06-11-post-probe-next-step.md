# Decision Brief — probe 이후 다음 스텝

> 목적: 정규화 probe 결과(V1 델타 `partial` — aes held-out서 naive 첫 격파, V2/V3 실패)를 받은
> 상태에서 다음 한 걸음 선택. 읽고 결정하면 그대로 실행. AWS 지출 옵션은 실행 직전 재동의.

## 현재 상태 (사실)

- **V1(델타 label)이 유일한 생존 축**: held-out aes 1.0824(naive 1.7198 대비 −37%), held-out gcd
  1.5723(naive 1.4117에 미달, winner 2.51 대비는 대폭 개선). [probe.md](../../../experiments/multidesign/probe/probe.md)
- **전이 비대칭**: gcd(53행)→aes 성공, aes(691행)→gcd 부분 실패 — 데이터 양보다 분포 방향이 지배.
  이 비대칭의 해명에는 **제3 설계**가 필요(2설계론 "방향" 변인을 분리 못 함).
- **AWS 스택은 destroy 상태**(2026-06-10) — ibex 실행 시 `cdk deploy` 재배포(수 분) 필요.
- probe 기계(변형 사본 + run_probe 드라이버)는 재사용 가능 상태로 커밋됨. 전 과정 결정적.

## 옵션 A — V1 조합 probe 확장 (무비용, 로컬)

- **내용**: V1×V3(델타 label + 무차원 feature) 사본 1~2개 추가 → 같은 LODO 2-fold 실측.
  spec §9가 예고한 후속("두 축 생존 시 조합" — V3은 죽었지만 V1 위 조합은 여전히 유효한 질문).
- **얻는 것**: V1의 gcd-side 미달(1.57 vs naive 1.41)을 조합이 메우는지 — transferable 승격 가능성.
- **Pros**: 비용 0, ~1시간 내 답. 기존 기계 그대로.
- **Cons**: **2-fold 정보 한계** — 같은 2설계에서 변형을 더 쥐어짜면 *이 2설계에 맞는* 조합을 고르는
  과적합 위험(probe의 다중비교). 근본 질문(비대칭·제3 분포)에는 답 못 함.

## 옵션 B — ibex 추가 → 3-fold 게이트 (AWS 비용)

- **내용**: `cdk deploy` 재배포 → ibex run-task(건별 동의) → prepare → gcd+aes+ibex 3설계 결합 →
  V1 포함 3-fold LODO 재실측 → `cdk destroy`.
- **얻는 것**: ① V1 축의 진짜 검증(훈련 fold에 2설계 혼합 — 양/음 slack 공존 훈련이 처음 성립)
  ② 전이 비대칭의 해명(제3 분포가 방향 변인 분리) ③ 교차설계 게이트의 3-fold 승격.
- **Pros**: 지금 정보 가치가 가장 큰 실험. V1 생존으로 기대 가치가 probe 전보다 상승(어제 보류
  사유였던 "정규화 미검증"이 해소됨).
- **Cons**: **AWS 실 과금**(ibex는 aes 74분보다 큼 — 수 시간 가능), flow 실패 마찰 위험(D5: 정지·보고),
  재배포 절차 필요.
- **비용**: Fargate task 1회(수 시간 × vCPU/메모리 단가 — 수 달러 수준 예상) + 재배포/destroy 무료.

## 옵션 C — 발견의 루프 환류 (구독만, spec 작업)

- **내용**: ① program.md에 V1 발견(델타 label·교차설계 약점)을 힌트로 추가 ② cross-design 지표를
  selection/승격 게이트에 편입하는 spec 작성(에이전트가 gen-004+에서 일반화를 스스로 최적화하게).
- **Pros**: AutoResearch 본령(Operator가 찾은 걸 에이전트가 증폭). 본 프로젝트 novelty 축(자동 연구
  프로세스)에 직접 기여. LLM 구독만 소모.
- **Cons**: 게이트 데이터가 아직 2설계 2-fold — **약한 지표를 게이트에 넣으면 그 지표에 과적합**하는
  에이전트가 나올 위험(gen-003 gaming 교훈). 3설계 확보 후가 안전.

## 옵션 D — 여기서 정지

- **내용**: probe 결과·기록까지. 다음 세션에서 결정.
- **Pros**: 지출 0. 발견은 전부 커밋됨.
- **Cons**: V1 모멘텀 미활용.

## 추천: B (ibex) — A를 ibex 대기 중 병렬로 (선택적)

근거: ① probe가 만든 결론 자체가 "제3 설계 없이는 비대칭·일반화 검증 불가"이고, V1 생존으로 ibex의
기대 정보 가치가 어제보다 명확히 상승 — "싼 검증 먼저" 패턴이 이번엔 *지출 정당화*로 닫힘.
② A는 무비용이지만 단독으론 2-fold 과적합 위험이 있어, **ibex 결과(3-fold)와 함께 읽을 때** 가치가
커짐 — ibex run-task가 도는 수 시간 동안 로컬에서 병렬 실행하면 일정 낭비 없음(독립 작업 병렬 원칙).
③ C는 3설계 데이터 확보 후 별도 brainstorm으로(약한 게이트 지표의 gaming 위험 회피).
